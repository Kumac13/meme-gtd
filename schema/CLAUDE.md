# Schema マイグレーションガイド

## ファイル命名規則

- 形式: `XXX_description.sql`（例: `007_add_calendar_datetime_fields.sql`）
- XXX: 3桁の連番（001から開始、既存の最大番号 + 1）
- description: スネークケースで機能を表現

## マイグレーションファイルの書き方

- 既存のマイグレーションファイルは変更せず、必ず新しい連番ファイルを追加する（適用済みDBとの整合が壊れるため）
- SQLファイルを置くだけでは適用されない。`packages/db/src/migrate.ts` の `migrations` 配列への登録が必要（ディレクトリは自動走査されない）
- 冪等に書く（`IF NOT EXISTS` を使用）
- `ALTER TABLE ... ADD COLUMN` を含むファイルには他のDDL・バックフィルを混在させない（列が既存だと duplicate column 例外でファイル全体が「適用済み」扱いになり、残りのSQLが実行されないまま記録されるため。トリガー再作成などは別の連番ファイルに分ける）
- 既存データの移行が必要な場合は同一ファイル内でUPDATE文も記述
- コメントで変更内容を説明

## 同期トリガー（014以降）

- `server_seq` / `uuid` の打刻は 014 のSQLiteトリガーが行う（CLIがAPIを経由せずDBに書くため、全書き込み経路を覆えるのはトリガーだけ）。アプリコードで `server_seq` を書かない
- issues / comments の更新トリガーは `AFTER UPDATE OF <列リスト>` で対象列を列挙している。同期対象にしたい列を追加するときは、新しいマイグレーションでトリガーを作り直して列リストに加える（列挙外の列だけを更新しても `server_seq` が増えず、オフラインクライアントに変更が届かない）

## 追随作業と本番適用

スキーマ変更の追随作業（shared型・リポジトリ・テスト・`docs/er-diagram.md`）と、
本番DBへの適用手順（ユーザーの明示的な指示があるときのみ・バックアップ必須）は **db-migration スキル** が唯一の正。
このディレクトリでSQLを書く前に、必ず同スキルのチェックリストを確認すること。
