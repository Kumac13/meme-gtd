# ドキュメントガバナンス

> 目的: 全ドキュメントのカタログと、追加・更新・削除のルール・書き方スタイルの定義
> 読むタイミング: ドキュメントを追加・変更・削除する前。どこに何を書くべきか迷ったとき
> 更新タイミング: ドキュメントの追加・削除・役割変更時（このカタログへの登録まで完了して作業完了）

## 追加・更新・削除のルール

1. **手順はスキルに書く**: 「何をどの順でやるか」は `.claude/skills/` のスキルが唯一の正。docs/ や CLAUDE.md に手順をコピーせず、スキル名で参照する（コピーは更新時に必ずズレて矛盾するため）
2. **事実は一箇所にだけ書く**: 同じ事実（データモデル・エンドポイント・環境変数など）を複数のドキュメントに書かない。正となる一箇所（上のカタログの「目的」列）に書き、他はポインタにする
3. **新規ドキュメントは最後の手段**: まず既存カタログのどれかに収まらないか検討する。作る場合は、このカタログへの登録と、ルート `CLAUDE.md` のルーティング表への追加（エージェントが読むべきものの場合）までを完了して作業完了とする
4. **ガバナンスヘッダーを付ける**: `docs/` 配下の各ドキュメントはH1直下に3行の引用ブロックを持つ:
   ```markdown
   > 目的: <1行>
   > 読むタイミング: <エージェント/人間がいつ開くべきか>
   > 更新タイミング: <どの変更で更新義務が生じるか>
   ```
5. **削除を恐れない**: どこからも参照されず更新もされないドキュメント、役割が他と重複したドキュメントは、統合・削除を提案する
6. **時点情報を書かない**: ロードマップ・カバレッジ率・「Last Updated」・特定バージョンでの状態説明は書かない（すぐ腐るため）。既知バグを文書に残す場合は検証日を明記し、修正時に削除する

## 書き方スタイル（Anthropicのプロンプトプラクティスに基づく）

CLAUDE.md・スキル・docs/ はエージェント（Claude Opus/Sonnet）のプロンプトの一部になる。以下に従う:

- **簡潔第一**: ルートCLAUDE.md は毎プロンプトに載る。1行増やす価値があるかを常に問う
- **強調は本番データ破壊リスクのみ**: `IMPORTANT` / `YOU MUST` / `<critical-safety>` は本番DB保護（ルートCLAUDE.md）と本番マイグレーション（db-migration スキル）だけに使う。最近のモデルは指示を文字通り従うため、強調の乱用は過剰反応を招き、本当に重要なルールの効果を薄める
- **肯定形 + 理由1行**: 「〜しない」より「〜する（理由）」。モデルは動機から適切に汎化する
  - 例: ×「FTS5を提案するな」→ ○「keyword検索は意図的にLIKEを採用（FTS5のunicode61が日本語の単語境界を扱えないため）。FTS5化の提案は不要」
- **構造化**: 番号付き手順（順序が重要な場合）・XMLタグ（`<critical-safety>` 等）で内容種別を分ける。具体例は1〜2個で十分
- **CLAUDE.md では太字強調と表を使わない**: 強調はプレーンな文で書く（強調の乱用防止と同じ理由）。表はこのカタログのような列挙データに限り、ルール記述には使わない（コンテキスト効率のため）
- **適用範囲を明示**: 「全コマンドで」「taskのみ」など、指示がどこまで及ぶかを書く（モデルは指示を勝手に広げも狭めもしない）
- **正確な件数・バージョン番号を書かない**: 「21ファイル」「v0.31.0〜」は腐る。「〜配下」「X.Y.Z」で書く
- **言語**: 日本語。コマンド・コード・識別子・ファイル名は英語のまま
- **コード例・コマンドは省略しない**: `[options]` や `...` で省略せず全文を記載する
