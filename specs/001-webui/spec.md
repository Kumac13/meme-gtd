# Feature Specification: Calendar View for Web UI

**Feature Branch**: `001-webui`
**Created**: 2025-11-25
**Status**: Draft
**Input**: User description: "WebUIにカレンダー画面を追加 - Task #102: 1日に何をどのくらいやったかわからない問題を解決するため、スケジュールされたタスクをカレンダー形式で表示する"

## Clarifications

### Session 2025-11-25

- Q: 表示対象は？ → A: Taskのみ（Memoは対象外）、status=Cancelは除外、scheduled_onが設定されているタスク
- Q: 表示形式は？ → A: 月/週/日の3種類を切り替え可能、デフォルトは今日を含む月表示
- Q: タスクの色分けは？ → A: Done=緑濃い色、それ以外=白
- Q: 時間の扱いは？ → A: start_timeあり→その時間帯に表示、start_timeなしor日跨ぎ→終日エリア
- Q: タスククリック時の動作は？ → A: 右側1/3にモーダル表示（GitHub Projectsのissue詳細風）、編集可能
- Q: 日付クリック時の動作は？ → A: 何もしない（新規作成は一旦対象外）
- Q: 月表示で多くのタスクがある場合は？ → A: 10件まで表示、それ以降はスクロール
- Q: URL状態管理は？ → A: nuqsを使用してクエリパラメータで管理
- Q: カレンダーライブラリは？ → A: @schedule-x/reactを調査推奨（要件を満たす場合）
- Q: タスク詳細モーダルの実装方針は？ → A: 既存ItemDetailを改修し、ページ/モーダル両対応にする
- Q: カレンダー用タスク取得のAPI戦略は？ → A: 既存GET /tasksにscheduledFrom/scheduledToパラメータを追加

## User Scenarios & Testing *(mandatory)*

### User Story 1 - View Tasks on Monthly Calendar (Priority: P1)

ユーザーは、特定の月にスケジュールされたタスクをカレンダー形式で一覧したい。これにより、1日に何をどのくらいやったか、今後の予定がどうなっているかを視覚的に把握できる。

**Why this priority**: カレンダー機能の最も基本的な価値を提供する。月表示があれば、ユーザーはスケジュール全体を俯瞰できる。

**Independent Test**: Calendarタブを開き、月表示でスケジュールされたタスクがカレンダー上に表示されることを確認する。各日付のセルにタスクタイトルがタイル表示され、Doneは緑、それ以外は白で色分けされている。

**Acceptance Scenarios**:

1. **Given** scheduled_onが設定された複数のTaskがある, **When** Calendarタブを開く, **Then** 今月のカレンダーが表示され、各日付にタスクタイルが表示される
2. **Given** 月表示のカレンダーを表示している, **When** ある日付に5つのタスクがある, **Then** 5つのタスクタイルがその日付セルに表示される
3. **Given** 月表示のカレンダーを表示している, **When** ある日付に15のタスクがある, **Then** 10件のタスクタイルが表示され、残りはスクロールで確認できる
4. **Given** status=doneのタスクがある, **When** カレンダーを表示する, **Then** そのタスクタイルは緑濃い色で表示される
5. **Given** status=next/waiting/somedayのタスクがある, **When** カレンダーを表示する, **Then** そのタスクタイルは白色で表示される
6. **Given** status=cancelのタスクがある, **When** カレンダーを表示する, **Then** そのタスクは表示されない

---

### User Story 2 - View Tasks on Weekly/Daily Calendar (Priority: P2)

ユーザーは、週表示や日表示でより詳細な時間軸でタスクを確認したい。start_timeが設定されたタスクは該当時間帯に表示され、終日タスクは上部の終日エリアに表示される（Google Calendarと同様のUI）。

**Why this priority**: P1の月表示で基本的な価値を提供した後、より詳細な時間管理を可能にする。週/日表示はstart_time/end_timeを活用した時間軸表示が必要。

**Independent Test**: 週表示・日表示に切り替え、start_timeのあるタスクが該当時間帯に表示され、終日タスクが上部エリアに表示されることを確認する。

**Acceptance Scenarios**:

1. **Given** 月表示を表示している, **When** 表示切替で「週」を選択する, **Then** 今日を含む週のカレンダーが時間軸付きで表示される
2. **Given** 月表示を表示している, **When** 表示切替で「日」を選択する, **Then** 今日のカレンダーが時間軸付きで表示される
3. **Given** start_time=09:00のタスクがある, **When** 週/日表示で確認する, **Then** 09:00の時間帯にタスクタイルが表示される
4. **Given** start_time=09:00, end_time=11:00のタスクがある, **When** 週/日表示で確認する, **Then** 09:00〜11:00の範囲でタスクタイルが表示される
5. **Given** start_timeがないタスクがある, **When** 週/日表示で確認する, **Then** 終日エリア（上部）にタスクタイルが表示される
6. **Given** 日を跨ぐタスク（scheduled_on=11/25, end_date=11/27）がある, **When** 週表示で確認する, **Then** 終日エリアで11/25〜11/27にまたがるタイルとして表示される

---

### User Story 3 - Navigate Calendar (Priority: P2)

ユーザーは、カレンダー上で異なる日付・期間に移動したい。「今日」ボタンで現在に戻り、前後ボタンで期間を移動できる。

**Why this priority**: ナビゲーションがないとカレンダーの実用性が大幅に低下する。P1と同時に必要だが、表示機能が先にないと意味がないためP2。

**Independent Test**: ナビゲーションボタンを使って前月・次月に移動し、「今日」ボタンで現在に戻れることを確認する。

**Acceptance Scenarios**:

1. **Given** 2025年11月の月表示を表示している, **When** 「次へ」ボタンをクリックする, **Then** 2025年12月の月表示に切り替わる
2. **Given** 2025年11月の月表示を表示している, **When** 「前へ」ボタンをクリックする, **Then** 2025年10月の月表示に切り替わる
3. **Given** 2025年10月の月表示を表示している, **When** 「今日」ボタンをクリックする, **Then** 今日を含む月の表示に切り替わる
4. **Given** 週表示を表示している, **When** 「次へ」ボタンをクリックする, **Then** 次の週の表示に切り替わる
5. **Given** 日表示を表示している, **When** 「次へ」ボタンをクリックする, **Then** 次の日の表示に切り替わる

---

### User Story 4 - View and Edit Task from Calendar (Priority: P3)

ユーザーは、カレンダー上のタスクタイルをクリックして詳細を確認し、その場で編集したい。GitHub Projectsのissue詳細のように、画面右側1/3にモーダルが表示される。

**Why this priority**: 閲覧だけでもカレンダーは使えるが、編集機能があると利便性が大幅に向上する。P1〜P3が実装された後の付加価値機能。

**Independent Test**: タスクタイルをクリックし、右側にモーダルが表示されること、モーダル内でタスクを編集・保存できることを確認する。

**Acceptance Scenarios**:

1. **Given** カレンダーにタスクタイルが表示されている, **When** タスクタイルをクリックする, **Then** 画面右側1/3にタスク詳細モーダルが表示される
2. **Given** タスク詳細モーダルが表示されている, **When** タスクのタイトルを編集して保存する, **Then** 変更がカレンダー上のタイルにも反映される
3. **Given** タスク詳細モーダルが表示されている, **When** タスクのステータスをDoneに変更する, **Then** カレンダー上のタイルの色が緑に変わる
4. **Given** タスク詳細モーダルが表示されている, **When** モーダル外をクリックする, **Then** モーダルが閉じる
5. **Given** タスク詳細モーダルが表示されている, **When** scheduled_onを変更する, **Then** カレンダー上でタスクが新しい日付に移動する

---

### User Story 5 - Persist Calendar State in URL (Priority: P3)

ユーザーは、カレンダーの表示状態（表示モード、日付、選択中のタスク）をURLで共有・ブックマークしたい。

**Why this priority**: URLに状態を持つことで、ブックマークや共有が可能になる付加価値機能。基本機能の後に実装。

**Independent Test**: カレンダーの状態を変更し、URLが更新されること、そのURLを開くと同じ状態が復元されることを確認する。

**Acceptance Scenarios**:

1. **Given** 月表示を表示している, **When** 週表示に切り替える, **Then** URLが`/calendar?view=week`のように更新される
2. **Given** URLが`/calendar?view=week&date=2025-11-25`, **When** そのURLを直接開く, **Then** 2025年11月25日を含む週表示が表示される
3. **Given** タスクモーダルを開いている, **When** URLを確認する, **Then** URLに`task=102`のようなパラメータが含まれる
4. **Given** URLが`/calendar?view=month&task=102`, **When** そのURLを直接開く, **Then** 月表示が表示され、タスク102のモーダルが開いた状態で表示される

---

### Edge Cases

- 1日に100件以上のタスクがある場合、パフォーマンスに影響はないか？（10件表示+スクロールで対応）
- scheduled_onが設定されていないタスクはどう扱うか？（カレンダーには表示しない）
- start_timeのみでend_timeがない場合は？（終日エリアに表示）
- end_dateがscheduled_onより前の不正データの場合は？（scheduled_onのみで表示）
- タイムゾーンの扱いは？（ブラウザのローカルタイムゾーンを使用）
- 日付クリックの動作は？（何もしない）
- カレンダーからの新規タスク作成は？（対象外）

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: システムは、WebUIのナビゲーションに「Calendar」タブを追加しなければならない
- **FR-002**: システムは、scheduled_onが設定されたTaskをカレンダー上に表示しなければならない
- **FR-003**: システムは、status=cancelのTaskをカレンダーから除外しなければならない
- **FR-004**: システムは、Memoをカレンダーに表示してはならない
- **FR-005**: システムは、月表示・週表示・日表示の3つの表示モードを提供しなければならない
- **FR-006**: システムは、デフォルトで今日を含む月表示を表示しなければならない
- **FR-007**: システムは、タスクをタイル形式で表示し、タイトルを表示しなければならない
- **FR-008**: システムは、status=doneのタスクを緑濃い色で表示しなければならない
- **FR-009**: システムは、status=done以外のタスクを白色で表示しなければならない
- **FR-010**: 週/日表示において、システムはstart_timeが設定されたタスクを該当時間帯に表示しなければならない
- **FR-011**: 週/日表示において、システムはstart_timeが設定されていないタスクを終日エリアに表示しなければならない
- **FR-012**: 週/日表示において、システムは日を跨ぐタスクを終日エリアに複数日にまたがる形で表示しなければならない
- **FR-013**: 月表示において、システムは1日あたり最大10件のタスクタイルを表示し、それ以上はスクロールで確認できるようにしなければならない
- **FR-014**: システムは、「今日」ボタンを提供し、クリックで今日を含む表示に戻れなければならない
- **FR-015**: システムは、「前へ」「次へ」ボタンを提供し、表示期間を移動できなければならない
- **FR-016**: システムは、表示モード切替ボタンを画面右上に配置しなければならない
- **FR-017**: システムは、タスクタイルクリック時に画面右側1/3にタスク詳細モーダルを表示しなければならない
- **FR-018**: タスク詳細モーダルは、タスクの編集機能を提供しなければならない
- **FR-019**: システムは、カレンダーの状態（表示モード、日付、選択タスク）をURLクエリパラメータに反映しなければならない
- **FR-020**: システムは、URLクエリパラメータからカレンダーの状態を復元できなければならない
- **FR-021**: GET /tasks APIは、scheduledFrom/scheduledToパラメータで日付範囲フィルタをサポートしなければならない

### Key Entities

- **Task**: カレンダー表示の対象。scheduled_on（表示日付）、start_time/end_time（時間帯）、end_date（終了日）、status（色分け）を使用
- **Calendar View State**: 表示モード（month/week/day）、基準日付、選択中のタスクID

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: ユーザーは、Calendarタブから2クリック以内で任意の月のスケジュール済みタスクを一覧できる
- **SC-002**: カレンダーの初期表示が2秒以内に完了する（100件のタスクがある場合）
- **SC-003**: 表示モード切替（月→週→日）が0.5秒以内に完了する
- **SC-004**: タスクタイルクリックからモーダル表示までが0.3秒以内に完了する
- **SC-005**: URLからカレンダー状態を復元した場合、同じ表示が100%再現される
- **SC-006**: 1日に50件のタスクがある場合でも、カレンダーのスクロールが滑らかに動作する
- **SC-007**: Google CalendarのUIに慣れたユーザーが、説明なしで基本操作（表示切替、ナビゲーション、タスク詳細表示）を行える

## Assumptions

- ブラウザのローカルタイムゾーンを使用する
- タスク編集モーダルは既存のItemDetailコンポーネントをページ/モーダル両対応に改修して使用
- カレンダーライブラリは調査結果を参考に選定（@schedule-x/reactを推奨）
- URL状態管理にはnuqsを使用（React Router v7対応確認済み）

## Technical Considerations (Informational)

調査結果に基づく技術選定の参考情報：

### カレンダーライブラリ候補
- **@schedule-x/react**: React 19完全対応、軽量、高カスタマイズ性、MIT無料
- FullCalendar: 実績あり、やや重い
- react-big-calendar: React 19対応未確認

### URL状態管理
- **nuqs**: React Router v7対応、型安全、5.5kB gzipped
