# Data Model: Web UI for meme-gtd

**Date**: 2025-10-20
**Feature**: Web UI for meme-gtd (Memos & Tasks Management)
**Branch**: `010-github-issue-meme`

## Overview

The Web UI is a **consumer** of the existing meme-gtd API. It does not define or manage its own data model. Instead, it uses TypeScript types generated from the API's OpenAPI specification (`/packages/api/docs/api/openapi.yaml`).

This document describes the **client-side representation** of API entities and how the UI interacts with them.

## Entity Definitions

All entities below are TypeScript interfaces generated by `openapi-typescript-codegen` from the OpenAPI spec. The UI treats these as read-only contracts defined by the API.

### Memo

**Source**: `GET /api/memos/{id}` response schema

**Description**: Represents a captured thought or note in the GTD capture phase. Memos contain freeform markdown content and can be promoted to tasks.

**Fields**:
| Field | Type | Nullable | Description |
|-------|------|----------|-------------|
| `id` | `number` | No | Unique identifier (primary key) |
| `type` | `"memo"` | No | Discriminator field, always "memo" |
| `bodyMd` | `string` | No | Markdown content of the memo (required, minLength: 1) |
| `title` | `null` | Yes | Always null for memos (only tasks have titles) |
| `status` | `null` | Yes | Always null for memos (only tasks have status) |
| `scheduledOn` | `null` | Yes | Always null for memos (only tasks have scheduled dates) |
| `isBookmarked` | `boolean` | No | Whether memo is bookmarked for quick access |
| `isDeleted` | `boolean` | No | Soft-delete flag (deleted memos filtered out by API) |
| `createdAt` | `string` (ISO 8601) | No | Timestamp when memo was created |
| `updatedAt` | `string` (ISO 8601) | No | Timestamp when memo was last updated |
| `labels` | `Label[]` | No | Array of labels assigned to this memo (detail view only) |
| `commentsCount` | `number` | No | Number of comments on this memo (detail view only) |

**State Transitions**: N/A (memos have no status field)

**Validation Rules**:
- `bodyMd` must be non-empty string (minLength: 1) when creating or updating
- Cannot set `title`, `status`, or `scheduledOn` on memos (API enforces)

**Relationships**:
- **Labels**: Many-to-many (memos can have multiple labels)
- **Links**: Many-to-many (memos can link to other memos/tasks)
- **Comments**: One-to-many (memo has many comments)

### Task

**Source**: `GET /api/tasks/{id}` response schema

**Description**: Represents an actionable item in the GTD workflow. Tasks have titles, statuses, and optional scheduled dates.

**Fields**:
| Field | Type | Nullable | Description |
|-------|------|----------|-------------|
| `id` | `number` | No | Unique identifier (primary key) |
| `type` | `"task"` | No | Discriminator field, always "task" |
| `title` | `string` | No | Task title (required, minLength: 1) |
| `bodyMd` | `string` | Yes | Markdown content of the task (optional) |
| `status` | `TaskStatus` | No | Current status (see TaskStatus enum below) |
| `scheduledOn` | `string` (ISO 8601 date) | Yes | Optional scheduled date (YYYY-MM-DD format) |
| `isBookmarked` | `boolean` | No | Whether task is bookmarked for quick access |
| `isDeleted` | `boolean` | No | Soft-delete flag (deleted tasks filtered out by API) |
| `createdAt` | `string` (ISO 8601) | No | Timestamp when task was created |
| `updatedAt` | `string` (ISO 8601) | No | Timestamp when task was last updated |
| `labels` | `Label[]` | No | Array of labels assigned to this task (detail view only) |
| `commentsCount` | `number` | No | Number of comments on this task (detail view only) |

**State Transitions**:
```
open → next → done
  ↓      ↓       ↑
waiting  ↓       ↑ reopen
  ↓      ↓       ↑
scheduled ↓      ↑
  ↓      ↓       ↑
  └──→ canceled ←┘
```

**Validation Rules**:
- `title` is required when creating tasks (minLength: 1)
- `status` must be one of: `open`, `next`, `waiting`, `scheduled`, `done`, `canceled`
- `scheduledOn` must be ISO 8601 date string (YYYY-MM-DD) if provided

**Relationships**:
- **Labels**: Many-to-many (tasks can have multiple labels)
- **Links**: Many-to-many (tasks can link to other tasks/memos)
- **Comments**: One-to-many (task has many comments)

### TaskStatus (Enum)

**Source**: OpenAPI schema enum

**Values**:
- `"open"` - Task is actionable and not yet started
- `"next"` - Task is the next action to take
- `"waiting"` - Task is waiting on someone/something
- `"scheduled"` - Task is scheduled for a future date
- `"done"` - Task is completed
- `"canceled"` - Task is canceled (will not be done)

**Status Actions (API Endpoints)**:
- `POST /api/tasks/{id}/close` → Sets status to `"done"`
- `POST /api/tasks/{id}/cancel` → Sets status to `"canceled"`
- `POST /api/tasks/{id}/reopen` → Sets status to `"open"`

### Label

**Source**: `GET /api/labels` response schema

**Description**: Categorization tag that can be attached to memos and tasks for organization.

**Fields**:
| Field | Type | Nullable | Description |
|-------|------|----------|-------------|
| `id` | `number` | No | Unique identifier (primary key) |
| `name` | `string` | No | Label name (unique, required, minLength: 1) |
| `description` | `string` | Yes | Optional label description |
| `createdAt` | `string` (ISO 8601) | No | Timestamp when label was created |

**Validation Rules**:
- `name` must be unique across all labels
- `name` is required when creating labels (minLength: 1)
- `description` is optional

**Relationships**:
- **Memos/Tasks**: Many-to-many (labels can be assigned to multiple memos/tasks)

**Important Constraint**: Deleting a label (`DELETE /api/labels/{name}`) removes it from **all** memos and tasks. The UI should warn users before label deletion.

### Link

**Source**: `GET /api/issues/{id}/links` response schema

**Description**: Represents a relationship between two issues (memos or tasks) with a specific type.

**Fields**:
| Field | Type | Nullable | Description |
|-------|------|----------|-------------|
| `id` | `number` | No | Unique identifier (primary key) |
| `sourceIssueId` | `number` | No | ID of the source issue (memo or task) |
| `targetIssueId` | `number` | No | ID of the target issue (memo or task) |
| `linkType` | `LinkType` | No | Type of relationship (see LinkType enum below) |
| `direction` | `"outgoing" \| "incoming"` | No | Direction relative to queried issue (computed by API) |
| `createdAt` | `string` (ISO 8601) | No | Timestamp when link was created |

**Validation Rules**:
- `sourceIssueId` and `targetIssueId` must exist in the database
- All three fields (`sourceIssueId`, `targetIssueId`, `linkType`) are required when creating links

**Relationships**:
- **Source Issue**: Belongs to one issue (memo or task)
- **Target Issue**: References one issue (memo or task)

**Direction Semantics**:
- **Outgoing**: Current issue is the source (I link to target)
- **Incoming**: Current issue is the target (Someone links to me)

**UI Display**: Links should be grouped by `linkType` and `direction` for clarity.

### LinkType (Enum)

**Source**: OpenAPI schema enum

**Values**:
- `"parent"` - Target is parent of source (source is child of target)
- `"child"` - Target is child of source (source is parent of target)
- `"relates"` - Bidirectional relationship (source and target are related)
- `"derived_from"` - Target is the original, source is derived from it

**Inverse Relationships** (for UI display):
- `parent` (outgoing) ↔ `child` (incoming)
- `child` (outgoing) ↔ `parent` (incoming)
- `relates` (outgoing) ↔ `relates` (incoming)
- `derived_from` (outgoing) ↔ source of (incoming, conceptual)

### Comment

**Source**: `GET /api/memos/{memoId}/comments` or `GET /api/tasks/{taskId}/comments` response schema

**Description**: Text note attached to a memo or task with markdown support.

**Fields**:
| Field | Type | Nullable | Description |
|-------|------|----------|-------------|
| `id` | `number` | No | Unique identifier (primary key) |
| `issueId` | `number` | No | ID of the parent issue (memo or task) |
| `bodyMd` | `string` | No | Markdown content of the comment (required, minLength: 1) |
| `createdAt` | `string` (ISO 8601) | No | Timestamp when comment was created |
| `updatedAt` | `string` (ISO 8601) | No | Timestamp when comment was last updated |

**Validation Rules**:
- `bodyMd` is required when creating comments (minLength: 1)

**Relationships**:
- **Issue**: Belongs to one issue (memo or task)

## Client-Side State Management

The UI manages the following **local state** (not persisted to API):

### UI State (Component-Level)

| State | Type | Scope | Purpose |
|-------|------|-------|---------|
| `loading` | `boolean` | Component | Indicates API request in progress |
| `error` | `Error \| null` | Component | Stores API error for display |
| `selectedStatus` | `TaskStatus \| "all"` | TasksPage | Task status filter selection |
| `bookmarkedFilter` | `boolean` | MemosPage, TasksPage | Bookmark filter toggle |
| `showLabelModal` | `boolean` | MemoDetail, TaskDetail | Label assignment modal visibility |
| `showLinkModal` | `boolean` | MemoDetail, TaskDetail | Link creation modal visibility |
| `showPromoteModal` | `boolean` | MemoDetail | Memo promotion modal visibility |
| `editingCommentId` | `number \| null` | CommentList | ID of comment being edited |

### Form State (Component-Level)

| State | Type | Scope | Purpose |
|-------|------|-------|---------|
| `bodyMd` | `string` | MemoForm, CommentForm | Memo/comment body input |
| `title` | `string` | TaskForm, PromoteModal | Task title input |
| `status` | `TaskStatus` | TaskForm, PromoteModal | Task status selection |
| `scheduledOn` | `string` | TaskForm | Task scheduled date input (YYYY-MM-DD) |
| `labelName` | `string` | LabelModal | New label name input |
| `labelDescription` | `string` | LabelModal | New label description input |
| `selectedLabelId` | `number \| null` | LabelModal | Selected label for assignment |
| `linkType` | `LinkType` | LinkModal | Link type selection |
| `targetIssueId` | `number \| null` | LinkModal | Target issue ID input |

**No Global State**: The UI uses local component state (useState) and custom hooks for API calls. No Redux/Zustand needed for MVP.

## API Contract Compliance

All TypeScript types are generated from OpenAPI spec to ensure compile-time contract validation:

```bash
# packages/web/package.json script
pnpm generate:api
# Runs: openapi-typescript-codegen --input ../api/docs/api/openapi.yaml --output ./src/api
```

**Generated Files**:
- `src/api/models/` - TypeScript interfaces for all entities (Memo, Task, Label, Link, Comment)
- `src/api/services/` - API client methods for all endpoints (MemosService, TasksService, etc.)
- `src/api/core/` - Fetch configuration and error handling

**Type Safety**:
- Compile errors if API payloads don't match schema
- IntelliSense autocomplete for all API methods and types
- Refactoring safety when API contracts change

## Constraints Summary

**API-Enforced**:
- Memo `bodyMd` required (minLength: 1)
- Task `title` required (minLength: 1)
- Label `name` unique across all labels
- Link `sourceIssueId` and `targetIssueId` must exist
- Deleted items (isDeleted=true) filtered out by API

**UI-Enforced**:
- Form validation before submission
- Confirmation dialogs for destructive actions (delete label, delete memo/task)
- Error messages for API validation failures (400, 409)

**Performance Constraints**:
- UI optimized for up to 500 combined memos/tasks
- Pagination not implemented in MVP (API returns all items)
